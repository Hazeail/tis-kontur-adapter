## Подтвержденные сценарии Контур

По локальной переписке и внешней документации сходится следующая картина.

### Сценарий 1. Мы сами формируем XML и подпись

Это канонический API-сценарий, в котором:

- TIS строит XML титула;
- TIS формирует CMS-подпись;
- TIS отправляет и XML, и подпись через API.

### Сценарий 2. Мы создаем черновик, а пользователь подписывает в UI Контур

Поддерживается как вспомогательный сценарий.

Важно:

- создание черновика и завершение подписания — это разные шаги;
- сам по себе черновик не равен завершенному этапу документооборота.

## Подтвержденные ограничения черновиков

Из переписки с Контур и из анализа API:

| Этап | Draft-сценарий |
|---|---|
| T1 | Допустим |
| T2 | Не является основным draft-сценарием |
| T3 | Допустим |
| T4 | Не является основным draft-сценарием |

Практический вывод:

- нельзя строить всю модель `T1 -> T2 -> T3 -> T4` как единый поток черновиков;
- `T2/T4` нельзя рассматривать как простое продолжение draft-потока `T1`.

## Каноническая модель Diadoc для ЭТрН

### T1

`T1` — это **первый титул**.

Канонический смысл:

- первый подписанный документ в цепочке;
- он должен быть отправлен как подписанный титул.

### T2 / T3 / T4

`T2`, `T3`, `T4` — это **ответные титулы**.

Канонический смысл:

- они отправляются не как новый первый документ;
- они привязываются к уже существующему документообороту;
- для них нужен контекст уже созданного и принятого первого титула.

## Ключевой инженерный вывод по текущей реализации

### Текущее поведение

В текущем коде `T1_INITIAL`:

- отправляет XML первого титула;
- получает `transportationId`, `titleId`, иногда `draftId`;
- сохраняет refs;
- считается успешным завершением T1.

### Что показывает UI Контур

После этого документ в UI Контур может оставаться в статусе:

- `Черновик`
- `Требует подписи`

### Что это значит

Это значит, что **текущая реализация, скорее всего, завершает T1 слишком рано**.

Иными словами:

- текущий код подтверждает создание документа;
- но еще не подтверждает завершение отправки подписанного первого титула.

## Почему это критично для T2

Если T1 не завершен как подписанный первый титул, то:

- T2 стартует на неполном состоянии документооборота;
- Контур может не считать T1 завершенным;
- отправка T2 приходит в неконсистентный внешний процесс;
- в результате на стороне Контур возникает немой `HTTP 500 / InternalServerError`.

## Подтверждающие признаки из практики

1. В логах TIS:
   - `T1_INITIAL` проходит `HTTP 200`;
   - `T2` падает в `500`.

2. В UI Контур:
   - документ виден;
   - черновик принят;
   - документ все еще требует подписи;
   - следующий бизнес-этап фактически не завершен.

3. В старых кейсах:
   - после полноценной подписи в UI Контур документ висит уже на следующем состоянии.

## Зафиксированный прикладной дефект на текущем контуре

На контрольном прогоне был получен конкретный серверный отказ Контур при подписании/отправке:

`MessageToPost.DocumentAttachments[0]: { Location = /Файл/Документ/СодИнфГО/СвВодит/@ИННФЛ, Message = ИНН 123456789012 не удовлетворяет контрольному соотношению }`

Что это означает:

- проблема была не в сертификате;
- проблема была не в transport context;
- в реконструкционном слое существовала собственная тестовая подстановка невалидного `ИННФЛ` водителя.

Источник дефекта локализован в:

- `Generated\Kontur\Integration\KonturT1ParticipantOverrideService.cs`

Сделанная коррекция:

- фиктивный `ИННФЛ = 123456789012` больше не подставляется;
- если ИНН водителя невалиден, он очищается;
- fallback теперь доводит блок `СвВодит` только через реквизиты водительского удостоверения;
- основной проект не затрагивался, правка сделана только в `7. Реконструкция`.

## Рабочая гипотеза реконструкции

Текущий адаптер нужно перестроить так, чтобы он явно различал:

1. создание документа / титула;
2. подписание титула;
3. отправку подписанного титула;
4. получение состояния, при котором разрешен следующий этап.

Это особенно важно для `T1`.

## Практический вывод для новой архитектуры

Новая модель должна быть такой:

### Для T1

`Build XML -> Sign -> Send signed first title -> Save refs -> Confirm T1 completed`

### Для T2

`Build XML -> Sign -> Resolve parent context -> Send recipient title -> Confirm T2 completed`

### Для T3 и T4

Тот же принцип, что и у ответных титулов:

`Build XML -> Sign -> Resolve docflow context -> Send recipient title`

## Что это означает для UI

UI должен отражать не просто набор кнопок, а явные состояния:

- XML не собран / собран;
- подпись не подготовлена / подготовлена;
- этап не отправлен / отправлен;
- этап не завершен / завершен;
- следующий этап заблокирован / доступен.

Именно под это потом должна вводиться модель `KonturStageState`.

## Что уже нельзя делать в новой модели

1. Нельзя считать `T1_INITIAL = T1 completed`.
2. Нельзя запускать `T2`, пока внешний процесс не показывает, что `T1` действительно завершен.
3. Нельзя смешивать:
   - создание черновика;
   - подписание черновика;
   - отправку подписанного титула;
   в один неявный шаг.

## Вывод для реконструкции

Главный архитектурный разрыв текущей реализации находится не только в UI и не только в XML, а в **неявной модели процесса T1**.

Поэтому реконструкция должна строиться вокруг:

- явного состояния этапа;
- отдельного use case для первого титула;
- отдельного use case для ответных титулов;
- тонкого UI-слоя, который показывает, что реально завершено, а что еще нет.
