# Контракт жизненного цикла T1

## Назначение

Документ фиксирует точную рабочую трактовку этапа `T1` для ТИС и реконструкционного слоя адаптера Контура.

Его задача:

- убрать двусмысленность между технической отправкой и бизнес-завершением;
- задать единый язык для UI, use case, SQL-состояния и ручного прогона;
- стать точкой истины для дальнейшей доработки completion-пути и открытия `T2`.

---

## 1. Граница ответственности T1

В контуре ТИС этап `T1` означает:

- подготовку первого титула;
- подписание именно текущего XML;
- отправку первого титула через адаптер Контура;
- фиксацию технического результата отправки;
- проверку внешнего признака перехода к следующему участнику;
- подтверждение завершения роли ТИС по первому титулу.

`T1` не означает автоматическое завершение всего документооборота в Контуре.

---

## 2. Обязательные состояния T1

### `XmlBuilt`

Означает:

- XML первого титула собран;
- XML относится к текущей версии этапа;
- XML можно подписывать.

Не означает:

- что XML уже подписан;
- что XML уже отправлен;
- что T1 завершен.

### `SignatureImported`

Означает:

- detached-подпись получена для текущего XML;
- подпись прошла локальную проверку against текущего XML;
- можно переходить к send-шагу.

Не означает:

- что T1 уже отправлен;
- что Контур принял подпись;
- что T1 завершен.

### `Sent`

Означает:

- адаптер выполнил операторный вызов отправки T1;
- операторный вызов завершился success-результатом;
- в состоянии этапа сохранены технические refs и диагностический результат.

Минимальные признаки `Sent`:

- `Sent = true`
- есть успешный результат отправки;
- сохранены или синхронизированы `TransportationId` и `TitleId`, если оператор их вернул.

Не означает:

- что роль ТИС по T1 завершена;
- что можно автоматически открывать `T2`;
- что внешний статус уже перешел в состояние следующего титула.

### `Completed`

Означает:

- роль ТИС по `T1` подтверждена как завершенная;
- внешний процесс дошел до признака, достаточного для открытия следующего титула;
- завершение подтверждено по evidence-модели и затем зафиксировано в состоянии этапа.

Не означает:

- что весь ЭТрН-документооборот Контур завершен;
- что дальнейших действий по документу больше не будет;
- что водитель, перевозчик или получатель уже завершили свои шаги.

### `NextStageAllowed`

Означает:

- следующий титул в модели ТИС может быть открыт;
- переход `T1 -> T2` разрешен именно в нашем процессе;
- это разрешение опирается на `Completed`, а не только на технический результат отправки.

Не означает:

- что следующий шаг уже выполнен;
- что можно игнорировать дополнительные проверки `T2`.

---

## 3. Запрещенные трактовки

Нельзя считать `T1` завершенным только по одному из признаков:

- HTTP 2xx от оператора;
- наличию `TransportationId`;
- наличию `TitleId`;
- появлению документа в UI Контура;
- смене локального `LastOperatorStatus`;
- ручной уверенности оператора без evidence-проверки.

Эти признаки могут быть достаточны для `Sent`, но недостаточны для `Completed`.

---

## 4. Единственно допустимая последовательность T1

Корректная последовательность шагов такая:

1. Собрать XML.
2. Импортировать подпись для этого XML.
3. Выполнить отправку T1.
4. Зафиксировать `Sent = true`.
5. Выполнить evidence-проверку завершения T1.
6. Если evidence достаточен, подтвердить завершение этапа.
7. Зафиксировать:
   - `Completed = true`
   - `NextStageAllowed = true`

Переход от шага 4 к шагу 6 без шага 5 является только временным ручным мостом и не должен считаться конечной production-моделью.

---

## 5. Единственно допустимое условие открытия T2

`T2` может стартовать только если для `T1` одновременно выполнено:

- `Sent = true`
- `Completed = true`
- `NextStageAllowed = true`

Дополнительно должно быть подтверждено:

- есть необходимый transport context;
- нет активной ошибки T1;
- evidence не указывает на draft или незавершенный внешний статус.

Если у `T1` есть только:

- `Sent = true`
- `TransportationId`
- `TitleId`

то `T2` запускать нельзя.

---

## 6. Временный мост и его границы

На текущем этапе проекта допускается временная ручная команда подтверждения `T1`.

Но ее границы такие:

- она не должна быть источником истины сама по себе;
- она должна опираться на результат `CheckT1CompletionUseCase`;
- в будущем она должна подтверждать уже проверенный completion, а не подменять его.

Иными словами:

ручной confirm допустим как операторское действие,
но недопустим как самостоятельное доказательство завершения `T1`.

---

## 7. Что это меняет в коде

Из этого контракта следуют обязательные правила для кода:

1. `SendT1...UseCase` имеет право выставлять `Sent`, но не имеет права автоматически выставлять `Completed`.
2. `CheckT1CompletionUseCase` имеет право сказать, что evidence достаточно, но не должен сам менять `Completed`.
3. `ConfirmStageCompletionUseCase` должен подтверждать только уже проверенный completion.
4. `KonturStageTransitionPolicy` должен считать `T2` допустимым только после `Completed` и `NextStageAllowed` у `T1`.
5. UI не должен интерпретировать `TransportationId` как разрешение на запуск `T2`.

---

## 8. Acceptance criteria

Контракт считается внедренным правильно, если:

1. В документации больше нет противоречий между `Sent` и `Completed`.
2. В коде `T2` не может стартовать по одному только факту успешной отправки `T1`.
3. Оператор видит distinction между:
   - отправлено;
   - завершено;
   - открыт следующий этап.
4. Evidence-проверка становится обязательной частью completion-пути.

---
