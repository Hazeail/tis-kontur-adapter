# Карта исходников и слоев реконструкции

## Назначение

Документ фиксирует, какие исходники были вынесены в реконструкционный слой и какую роль они выполняют в текущем контуре.

Это нужно, чтобы:

- не потерять границы ответственности;
- понимать, какие файлы будут меняться в реконструкции;
- видеть, что уже можно переиспользовать, а что нужно переразложить.

## Корневые разделы исходников

| Путь | Назначение |
|---|---|
| `Исходники\Account\EpdSrc` | WebForms UI и code-behind для рабочего экрана и подписи |
| `Исходники\Generated` | Генераторы титулов и прямые code-dependency вне папки `Generated\Kontur` |
| `Исходники\Generated\Kontur` | Контур адаптера, модели, интеграционные сервисы, storage |
| `Исходники\Modules` | Legacy-модули генерации ЭТрН, используемые builder-слоем |
| `Исходники\Sql\KonturIntegration` | SQL-контур интеграции в `Perdoc` |

## Главные проблемные зоны текущего контура

### 1. `KonturProbe.aspx.cs`

Проблема:

- слишком много экранной, сценарной и интеграционной логики в одном code-behind.

Следствие:

- трудно тестировать;
- трудно менять порядок процесса;
- велика цена любой правки.

### 2. `KonturStageRunner.cs`

Проблема:

- в одном месте смешаны:
  - построение XML;
  - подготовка temp-файлов;
  - выбор подписанта;
  - сохранение артефактов;
  - запуск этапа.

Следствие:

- orchestrator превращается в class-god;
- сценарий трудно разделить на независимые шаги.

### 3. `KonturAdapter.cs`

Проблема:

- в одном адаптере смешаны разные API-сценарии;
- логика `T1` и логика ответных титулов еще не разведены до конца.

Следствие:

- сложно выстроить канонический процесс:
  - `T1` как первый титул;
  - `T2/T3/T4` как ответные титулы.

### 4. Жесткая зависимость от legacy-builder ЭТрН

Проблема:

- адаптер Контур уже использует `ETRNtituls`, но этот слой живет вне папки `Generated\Kontur`;
- без копии этих файлов реконструкция была бы неполной.

Следствие:

- при проектировании новой архитектуры нужно явно учитывать границу:
  - адаптер Контур не должен дублировать маппинг TIS;
  - но должен изолировать вызов legacy-builder внутри отдельного use case / builder-порта.

## Вывод для реконструкции

Реконструкция не должна переписывать все с нуля.

Нужно сохранить:

- `KonturClient`
- `Repository`-контур
- XML-сервисы
- `Kontur-only`

И постепенно перевести архитектуру к модели:

`UI -> ScreenService -> UseCase -> Gateway/Repository -> Adapter/DB`
