# Фиксация выполнения T1 в реконструкции Контур ЭТрН

## Назначение

Документ фиксирует текущее понимание выполнения `T1` в реконструкционном слое:

- как `T1` собирается, подписывается и отправляется сейчас;
- что уже перенесено в новую архитектурную модель;
- что пока остается недоказанным;
- какие риски по `T1` блокируют окончательный перевод всей цепочки `T1 -> T2 -> T3 -> T4`.

Документ нужен как отдельная опорная точка, потому что именно `T1` является главным архитектурным разрывом текущего адаптера.

## Главный вывод по T1

`T1` нельзя считать завершенным только по факту успешного вызова `T1_INITIAL`.

Текущее рабочее предположение реконструкции:

- `T1_INITIAL`, вероятно, создает документ или черновик;
- это еще не гарантирует завершение отправки подписанного первого титула;
- полноценное завершение `T1` должно отличаться от технического факта создания внешнего документа.

Именно поэтому в реконструкции уже зафиксирована модель:

- `T1.Sent`
- `T1.Completed`
- `T1.NextStageAllowed`

а не только бинарное представление "отправлено / не отправлено".

## Как T1 устроен в целевой модели

Целевая цепочка `T1` должна выглядеть так:

1. Сформировать XML первого титула.
2. Подготовить корректную detached-подпись для этого XML.
3. Отправить подписанный первый титул в Контур.
4. Зафиксировать технический факт отправки как `Sent = true`.
5. Отдельно подтвердить, что этап действительно бизнес-завершен.
6. Только после этого разрешать следующий этап:
   - `Completed = true`
   - `NextStageAllowed = true`

## Что уже реализовано в реконструкционном слое

### 1. Build T1

Реализовано:

- `BuildStageTitleUseCase`
- нормализация T1 через:
  - `KonturEtrnT1234XmlService`
  - `KonturT1ParticipantOverrideService`

Что это дает:

- T1 собирается не напрямую из UI-логики;
- XML T1 можно сохранить как явный stage-артефакт;
- пересборка XML сбрасывает старые признаки подписи и отправки.

### 2. Import signature for T1

Реализовано:

- `ImportStageSignatureUseCase`

Что это дает:

- detached CMS проверяется до фиксации состояния;
- подпись сохраняется как stage-артефакт;
- состояние этапа получает `SignatureImported = true`.

### 3. Send T1

Реализовано:

- `SendT1FirstTitleUseCase`
- `IKonturFirstTitleGateway`
- adapter-backed gateway поверх текущего адаптера
- `SyncStageStateRefsUseCase` для подтягивания `TransportationId` и `TitleId` из `TEpdOperatorRef` в `KonturStageState`

Что уже зафиксировано в коде:

- успешная отправка T1 фиксирует только `Sent = true`;
- успешная отправка T1 через новый send-path сохраняет `TransportationId` и `TitleId` в явном состоянии этапа;
- успешная отправка T1 через legacy runtime сразу синхронизирует `TransportationId` и `TitleId` из `TEpdOperatorRef` в явное состояние этапа;
- use case намеренно не переводит `T1` в `Completed`;
- use case намеренно не разрешает следующий этап автоматически.

Это важная и осознанная граница реконструкции.

### 4. Manual completion bridge

Реализовано:

- `ConfirmStageCompletionUseCase`

Что это дает:

- оператор может явно подтвердить завершение этапа;
- только после этого этап получает:
  - `Completed = true`
  - `NextStageAllowed = true`

Пока это временный мост, а не финальная модель backend-подтверждения.

## Что уже подключено к UI по T1

В `KonturProbe.aspx.cs` уже есть:

- сборка T1 XML;
- ручной импорт подписи;
- экранное чтение summary через новый screen-service;
- bridge совместимости между reconstruction state и legacy runtime;
- ручное подтверждение завершения этапа оператором.

Что пока не сделано:

- новый T1 send-path не включен как обязательный runtime для страницы;
- старый путь не выкинут;
- окончательная семантика завершения T1 не доказана.

## Что именно считается доказанным по T1 сейчас

На текущий момент доказано следующее:

1. `T1` требует отдельной модели жизненного цикла.
2. Успешный runtime-вызов и завершенный этап — это разные вещи.
3. Следующий этап нельзя открывать автоматически только по факту технической отправки.
4. Для `T1` уже нужен явный слой состояния, иначе `T2` стартует слишком рано.

## Что по T1 пока не доказано

Остается недоказанным главный вопрос:

**какой именно API-вызов или набор вызовов Контур считает полноценной отправкой подписанного первого титула, а не только созданием документа/черновика.**

Пока этот вопрос не закрыт, нельзя:

- считать `T1_INITIAL` окончательным успешным выполнением `T1`;
- переводить UI целиком на новый runtime без предохранителей;
- объявлять реконструкцию `T1` завершенной.

## Зафиксированный дефект T1, найденный на практическом прогоне

На ручной проверке был получен конкретный отказ Контур:

`MessageToPost.DocumentAttachments[0]: { Location = /Файл/Документ/СодИнфГО/СвВодит/@ИННФЛ, Message = ИНН 123456789012 не удовлетворяет контрольному соотношению }`

Что удалось установить:

- проблема находилась в реконструкционном слое;
- в `KonturT1ParticipantOverrideService` существовала тестовая подстановка невалидного `ИННФЛ` водителя;
- это значение ломало серверную валидацию Контур еще до подтверждения правильной модели `T1`.

Сделанная правка:

- фиктивный `ИННФЛ = 123456789012` больше не подставляется;
- невалидный ИНН водителя очищается;
- fallback блока `СвВодит` теперь опирается только на реквизиты водительского удостоверения.

## Что это означает для повторной проверки T1

После правки старые уже собранные XML не исправляются автоматически.

Для корректной проверки `T1` нужен полный повторный цикл:

1. Пересобрать T1 XML.
2. Заново подписать именно новый XML.
3. Повторно импортировать новый `.sgn`.
4. Повторить отправку T1.

Без этого будет использоваться старый XML со старым содержимым блока водителя.

## Текущее состояние T1 в терминах реконструкции

| Область | Состояние |
|---|---|
| Build XML | Вынесен в use case и подключен к реконструкционному слою |
| Signature import | Вынесен в use case и работает add-only |
| Stage state | Явная модель уже есть |
| Screen read-model | Уже умеет читать явное состояние |
| Send use case | Создан |
| Completed / NextStageAllowed | Есть как явная модель и временный операторский мост |
| Семантика полноценного завершения T1 | Не подтверждена окончательно |

## Что запрещено считать по T1

Нельзя считать равными вещи:

- `T1_INITIAL` и `T1 completed`;
- `черновик создан` и `первый титул завершен`;
- `есть transportationId` и `можно запускать T2`;
- `операторный вызов успешен` и `следующий этап уже разрешен`.

## Практический сценарий проверки T1

Рекомендуемый текущий ручной сценарий:

1. Выбрать `TimelineId = 10212`.
2. Сформировать новый `T1`.
3. Подписать новый `t1_override...xml`.
4. Импортировать актуальный `.sgn`.
5. Отправить `T1`.
6. Проверить:
   - исчезла ли ошибка по `СвВодит/@ИННФЛ`;
   - какой статус документа показывает Контур;
   - остается ли документ черновиком;
   - требует ли Контур дополнительного шага подписи/подтверждения.
7. Только после этого оценивать, можно ли подтверждать этап как `Completed`.

## Что должен сделать следующий инженерный шаг

Следующий шаг по `T1` должен ответить не на вопрос "как еще отправить XML", а на вопрос:

**какой внешний признак или какой API-результат действительно означает, что первый титул завершен как подписанный документ и открывает право на `T2`.**

До ответа на этот вопрос реконструкция `T1` считается продвинутой, но не закрытой.
